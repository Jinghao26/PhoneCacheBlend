PhoneCacheBlend — Production Journal
=====================================
Started: 2026-06-16
Device:  iPhone 16, 128GB storage, 8GB RAM
Runtime: llama.cpp + Metal (Swift app, based on llama.swiftui)
Model:   Qwen2.5-1.5B-Instruct, Q4_K_M (4-bit)

Goal
----
Deploy on-device RAG with KV chunk reuse to improve efficiency (TTFT,
power, heat) compared to full prefill on every query.

First milestone (v0.1): simple reuse by chunk ID + GRAPH fuse
  - Prefill each document chunk once, save KV state to disk keyed by chunk_id
  - On a new query, retrieve chunk IDs, load cached KV, stitch, prefill only
    the question suffix, fuse (GRAPH), then decode
  - CacheBlend GRAPH fusion validated on device (Phase E, 2026-06-17)

Deferred to later milestones
  - Q_sub gather in CHECK Step C (use Step A Q without recompute; Metal buffer issue)
  - Cache system prefix as its own chunk_id (fixed template, reusable per model)
  - Stitch hot path: skip disk reload for HIT chunks (RAM cache or in-session KV)
  - Full LMCache-style cross-session production cache
  - Qwen 3B upgrade if 1.5B quality is insufficient


Background (design context)
---------------------------
Traditional RAG concatenates retrieved passages and runs one full prefill:

  [system_prefix + chunk_A + chunk_B + ... + question] → decode

Cost scales with total prompt length. Every query pays full prefill even
when passages were seen before.

CacheBlend (reference repo: ~/Desktop/CacheBlend) proposes:
  1. Collect — prefill each chunk in isolation, save per-layer K/V
  2. Stitch  — concatenate cached K/V tensors
  3. Fuse    — selectively recompute ~16% of tokens + full question suffix
               to fix cross-chunk attention errors

Key insight: chunks prefilled alone never "saw" earlier chunks (causal
attention). Stitching is an approximation; fusion corrects the worst errors.
For mobile v0.1 we skip fusion and accept approximate stitch + question
prefill, with optional boundary recompute added later if quality drops.

What "previous prefill" means
  - NOT automatic reuse from the last full user prompt
  - NOT built-in llama.cpp prefix caching alone
  - IS: reuse KV saved for a specific chunk_id from an earlier prefill
        (offline index or prior query in the same session)

RAG chunk identity
  - Two similar questions may retrieve overlapping chunks (e.g. same A, B)
  - Chunks are keyed by content hash / stable chunk_id, not by question
  - Same chunk_id across requests → same cached KV blob on disk


KEY NOTE — PCB vs server CacheBlend: not comparable for consistency / TTFT
-------------------------------------------------------------------------
Server CacheBlend (~/Desktop/CacheBlend, blend_wikimqa.py / cache_blend_core.py)
and PhoneCacheBlend measure different things. Do NOT use paper CB TTFT or our
HPC quality-suite TTFT to verify that PCB implements CB “correctly” or to expect
matching speedup numbers.

What server CB benchmarks actually time (WikiMQA / Musique / SAMSum paper table):
  1. Collect (excluded from reported “TTFT cache”):
     - Chunk-wise GPU prefill per passage; stitch K/V into model.old_kvs in VRAM
     - Done every query; no persistent disk/RAM blob cache in example scripts
  2. “TTFT cache” (what the paper reports, e.g. WikiMQA 56 ms):
     - TTFT of the final llm.generate() only: check=True, fuse with old_kvs
     - first_token_time − first_scheduled_time on that single call
  3. “TTFT full” (e.g. 190 ms):
     - TTFT of one-shot full prefill on the same stitched prompt (check=False)
  4. “Speedup” (e.g. 3.4×):
     - TTFT_full / TTFT_cache — fuse decode vs monolithic prefill decode ONLY
     - Collect cost is NOT in numerator or denominator

What PhoneCacheBlend measures (E2E TTFT):
  - ensure (disk/RAM cache HIT or GPU collect on MISS)
  - stitch (deserialize / memcpy / optional label GPU)
  - fuse (GRAPH cache_blend_fuse)
  - question post-fuse + first answer token
  - Plus: disk FIFO (~18 GB WikiMQA corpus) + RAM hot cache (64 slots) — CB
    example scripts do not have this tier; LMCache hooks in vllm_blend are off

Implication:
  - Latency / speedup: apples-to-oranges. PCB E2E will look much slower than
    paper “TTFT cache” even when fuse kernel is correct; warm PCB vs cold CB
    collect is also unfair.
  - Quality (F1 / key-phrase): loosely comparable on same passages/questions
    (fuse vs full prefill answer), not proof of identical timing methodology.
  - To compare implementations: same metric on each platform separately (see
    CacheBlend_vs_PhoneCacheBlend_comparison.txt Part 2), or add server-side
    E2E timing that includes collect + fuse (not in upstream scripts today).

Reference: comparison log Part 1 (paper table) = fuse-step TTFT only; Part 2 =
per-platform Simple/Harder matrix with explicit “not cross-platform comparable”.

Architecture (v0.1 — chunk ID reuse)
------------------------------------

  Corpus (offline)
    documents → split into chunks → assign chunk_id = hash(model_id + chunk_text)
    for each chunk_id:
      prefill chunk tokens (positions 0..n-1)
      save llama_state_seq_save_file("cache/{chunk_id}.bin")

  Query (online)
    question → retriever → [chunk_id_A, chunk_id_B, ...]
    stitch:
      pos = 0
      for each chunk_id in order:
        load cache/{chunk_id}.bin into scratch sequence
        copy KV to main sequence at [pos, pos + n_tokens) via seq_cp
        pos += n_tokens
    prefill question suffix at n_past = pos
    decode answer

  Cache miss
    prefill chunk → save to cache/{chunk_id}.bin → continue stitch

  chunk_id schema (v0.1)
    chunk_id = SHA256(model_version + tokenizer_version + chunk_text)
    metadata sidecar: cache/{chunk_id}.json
      { "chunk_id", "tokens", "n_tokens", "model", "created_at" }


Prompt layout
-------------
  [system_prefix] + [chunk_A] + [chunk_B] + ... + [question_suffix]

  system_prefix: fixed chat template / RAG instructions
                 (today: GPU prefill every stitch; future: own chunk_id "system_v1")
  question_suffix: always full prefill (never cached in v0.1)

Phase B — Chunk KV store (COMPLETE)
------------------------------------

Goal (confirming intent)
  YES — Phase B is about saving KV for document chunks and reusing them on
  later queries when the same chunk_id appears again.

  Phase A: every query pays full prefill on [prefix + all passages + question].
  Phase B: pay full prefill per chunk ONCE, then on later queries load cached
  KV from disk instead of re-prefilling those passages.

  We are NOT doing CacheBlend fusion yet. Reuse = load saved state + stitch +
  prefill question only (Phase C wires the full query path; Phase B proves
  save/load and cache hit/miss).

What is a "chunk"?
  - One passage block from the RAG UI (text between --- separators), OR
  - A fixed system-prefix chunk (optional separate chunk_id "system_v1")
  - NOT the question — question is always fresh prefill in v0.1

What is chunk_id?
  chunk_id = SHA256(model_filename + chunk_text)
  Same text + same model → same id → same cache file on disk.
  Two different questions that retrieve the same passage → same chunk_id → reuse.

Disk layout (on device)
  Documents/chunks/
    {chunk_id}.bin          llama_state_seq_save_file (KV + state)
    {chunk_id}.json         metadata: tokens[], n_tokens, model, created_at

When KV gets saved (collect)
  Trigger: first time a passage is used in any query (cache miss).
  Steps:
    1. Tokenize chunk text (with same wrapper as query path)
    2. llama_decode() on chunk alone (seq 0, positions 0..n-1)
    3. llama_state_seq_save_file → Documents/chunks/{chunk_id}.bin
    4. Write sidecar .json

When KV gets reused (cache hit)
  Trigger: a later query includes a passage whose chunk_id already has .bin on disk.
  Steps:
    1. Skip prefill for that passage
    2. llama_state_seq_load_file into scratch sequence
    3. (Phase C) stitch into main sequence at correct position
    4. Prefill only question suffix
    5. Decode answer

Cold vs warm query (important for metrics)
  Cold (all cache miss):
    - Every chunk prefilled + saved
    - SLOWER than Phase A baseline (extra save I/O + per-chunk prefills)
    - Expected; amortize over future queries

  Warm (all cache hit):
    - Load .bin files from disk, stitch, prefill question only
    - TTFT should be LOWER than Phase A (~768 ms prefill on 119 tokens)
    - This is the win we are building toward

  Mixed (partial overlap):
    - Reuse hits, prefill+save only missing chunks
    - Common in real RAG (similar questions, overlapping passages)

Phase E — CacheBlend fuse (status 0 / 1 / 2 → llama.cpp) [IMPLEMENTED — see results below]
--------------------------------------------------------
Goal: after Phase C stitch + HKVD imp_indices, run fused prefill then decode.
Maps CacheBlend temp_status to llama.cpp graph + KV cache behavior.

Enum (proposed, llama.h):
  LLAMA_FUSE_STATUS_NONE   = -1   // normal decode (1 token)
  LLAMA_FUSE_STATUS_FULL   =  0   // CacheBlend status 0
  LLAMA_FUSE_STATUS_CHECK  =  1   // CacheBlend status 1
  LLAMA_FUSE_STATUS_SUBSET =  2   // CacheBlend status 2

Config struct (proposed):
  llama_cache_blend_params {
    check_layer;      // e.g. 1 (0-based block index)
    recomp_ratio;     // e.g. 0.18
    suffix_len;       // question token count (always in imp)
    n_imp;            // filled after check, or supplied if precomputed
    imp_indices[];    // sorted token positions [0, n_prompt)
    mask_mode;        // POSITION_CAUSAL (default) or BOTTOM_RIGHT
  }

Prerequisites (already have):
  - Stitched KV in llama_kv_cache seq 0, positions [0, n_prompt)
  - kv_unified = true, n_seq_max >= 2 for stitch collect path
  - llama_compute_hkvd_indices for CHECK layer (or pass precomputed imp)

Per-layer assignment (check_layer = k):
  layers [0 .. k-1]     → FULL
  layer  k              → CHECK
  layers [k+1 .. L-1]   → SUBSET
  after fuse prefill    → NONE (normal decode)

--- STATUS 0: LLAMA_FUSE_STATUS_FULL (CacheBlend status 0) ---
  When: layers before check_layer; also optional “warmup” if check_layer=0 skipped.
  Hidden stream:
    ubatch.n_tokens = n_prompt (N)
    ubatch.pos[i] = i; token[i] from stitch token list
    All N rows through input LN → QKV → attn → MLP → next layer
  KV cache:
    Normal llama_kv_cache cpy_k / cpy_v for all tokens in batch (write fresh K/V)
    Do NOT read stitched old_kv for forward hidden path (fresh block compute)
  Attention mask:
    Standard causal: set_input_kq_mask, n_kv == n_tokens == N
  HKVD: none
  Output to next layer: h_out shape N × n_embd

--- STATUS 1: LLAMA_FUSE_STATUS_CHECK (CacheBlend status 1) ---
  When: layer index == check_layer (default k=1).
  Hidden stream IN: N rows (from FULL layer below).
  Step A — full-width QKV (all N):
    Q, K, V = proj(h[0..N-1]); apply RoPE at ubatch.pos[i]
  Step B — HKVD (once, at this layer):
    v_old = read V from stitched KV at positions [0,N)  (live cache / export)
    v_new = V just computed in Step A
    imp_indices = llama_compute_hkvd_indices(v_old, v_new, ..., suffix_len, ratio)
    OR use precomputed imp_indices from Swift probe (skip v_new full prefill)
  Step C — subset attention:
    ubatch.n_tokens = n_imp = |imp_indices|
    ubatch.pos[j] = imp_indices[j];  // original absolute positions
    Q_sub = Q[imp_indices]
    K_attn, V_attn = full cache get_k/get_v (length n_kv = N)
    Patch: cpy_k/cpy_v fresh K/V into cache cells at imp_indices only
    Optional: re-RoPE stitched K at non-imp slots if stitch rotation wrong
  Attention mask:
    n_tokens = n_imp, n_kv = N (same as decode: subset Q vs full KV)
    POSITION_CAUSAL: query at pos p sees keys with cell pos <= p
    (or BOTTOM_RIGHT to match vLLM CacheBlend exactly)
  Hidden stream OUT: n_imp rows only
    h_out = h[imp_indices] after attn+MLP on subset stream
    Store imp_indices + n_imp in fuse params for upper layers

--- STATUS 2: LLAMA_FUSE_STATUS_SUBSET (CacheBlend status 2) ---
  When: layers > check_layer.
  Hidden stream:
    ubatch.n_tokens = n_imp
    ubatch.pos[j] = imp_indices[j]
    QKV proj only on n_imp rows
  KV cache:
    Read full K/V tensors (n_kv = N) for attention
    After computing fresh K/V for imp rows: patch only imp_indices slots
    Non-imp positions: leave stitched K/V unchanged
  Attention mask: same as CHECK (n_imp queries vs N keys)
  Hidden stream OUT: n_imp rows
  Repeat for each layer until final norm → logits from last suffix token

--- STATUS -1: LLAMA_FUSE_STATUS_NONE (decode) ---
  Normal llama_decode: 1 token, n_kv grows; no fusion.

Key llama.cpp touchpoints:
  llama-context.cpp     — fuse prefill driver (layer loop, pick status per il)
  llama-graph.cpp       — build_attn with n_tokens != n_kv; fuse mask fill
  llama-kv-cache.cpp    — patch k/v at cell indices; set_input_kq_mask fuse variant
  llama.h               — llama_cache_blend_params, llama_decode_fused(...)
  LibLlama.swift        — after stitch: llama_cache_blend_fuse(...); then decode

Entry flow (PhoneCacheBlend):
  1. ensurePassagesCached + completion_init_with_cached_chunks (stitch, KV full N)
  2. optional probe → imp_indices (or compute inside CHECK at layer k)
  3. llama_cache_blend_fuse(ctx, params)  // replaces extra full prefill + naive decode KV
  4. completion_loop (decode)

Invariants:
  - KV buffer length always n_prompt (N); never shrink to n_imp
  - imp_indices includes all suffix positions [N-suffix_len .. N-1]
  - Hidden activations after check: n_imp rows only
  - check_layer k requires layers 0..k-1 as FULL (all N)

Phase E — CacheBlend fuse (COMPLETE)
------------------------------------
Goal: after Phase C stitch, run CacheBlend-style fuse then decode (no extra full prefill).

API (llama.h):
  llama_cache_blend_fuse(ctx, &params)
  llama_cache_blend_params {
    mode;           // TOKEN_RECOMPUTE | GRAPH
    check_layer;    // default 1
    recomp_ratio;   // default 0.18
    suffix_len;     // question token count
    tokens[];       // full stitch token list (must match KV positions)
    imp_indices[];  // optional precomputed; else HKVD inside fuse
  }

Fuse modes:
  TOKEN_RECOMPUTE (v1 fallback):
    seq_rm from min(imp) → one contiguous llama_decode of tokens[min_imp..N)
    through ALL layers. Simple, causally correct, ~491/512 tokens recomputed.
    Device fuse ~2144 ms (6-chunk Scaling test, pre-GRAPH).

  GRAPH (default, production):
    Multi-pass CacheBlend status 0/1/2 at check_layer k=1:
      FULL:       layers [0 .. k-1] on all N tokens (chunked by n_batch)
      CHECK (A):  QKV+RoPE on all N at layer k; write KV; export V for HKVD
      HKVD:       v_old (stitched) vs v_new (Step A); pick ~18% prefix + suffix
      CHECK (C):  subset attn on |imp| at layer k (n_imp queries vs N KV)
      SUBSET:     layers [k+1 .. L-1] on |imp| scattered tokens only
    Target: only ~106 tokens in upper layers vs ~491 in TOKEN_RECOMPUTE.

Open risks
----------
  - Cross-chunk attention: chunks prefilled in isolation; stitch is approximate
    until fusion / selective recompute (HKVD indices identify worst positions)
  - RoPE / position mismatch when stitching chunks prefilled at local pos 0..n-1
    into global positions; may need absolute-position prefill or boundary
    recompute before fusion
  - seq_cp with kv_unified=false + n_seq_max=2: cross-stream copy WIPES stream 0
    on each chunk merge — FIXED by kv_unified=true (use unified KV buffer)
  - Chat template must be identical between collect and query paths
  - First query per new chunk is always slower (collect cost)
  - HKVD probe overhead when enableHkvdProbe=true (~1 s full prefill per query)


Future goals (refinements)
--------------------------
  [ ] System prefix cache — treat PhoneCacheBlendConfig.systemPrefix as a fixed
      chunk_id (e.g. SHA256(model + "system_v1")), save/load like passages;
      stitch order: [cached prefix] → [chunks] → [question]. Small latency win.
  [x] HKVD fusion — GRAPH multi-pass fuse wired (Phase E, 2026-06-17)
  [ ] RAM hot cache — keep HIT chunk KV in memory to avoid disk reload each stitch
      (~540–550 ms stitch prefill today reloads all .bin files regardless of HIT)
  [ ] Question prefill after fuse (not during stitch) —
      Today: stitch prefills labels + question (~27 tok), then fuse runs on all N
      tokens with suffix_len = question length; fuse always recomputes the full
      question suffix (~18% prefix + all suffix in |imp|). That double-touches the
      question and adds cost to the GRAPH pass (layer 0 on N, HKVD on N, subset
      recompute on suffix tokens).
      Goal: stitch passage-only KV (prefix + [n] labels + cached chunk bodies),
      fuse with suffix_len = 0 (HKVD / recompute on passage boundaries only),
      then normal GPU prefill of question suffix on fused KV before decode.
      Rationale: CacheBlend fixes stitched passage consistency, not the query;
      question was never cached; causal attention lets post-fuse question prefill
      see corrected passage KV. Labels stay in stitch (~2–4 tok/chunk, negligible).
      Expected: modest fuse savings (fewer tokens in |imp|, no suffix in GRAPH);
      stitch dominated by KV merge unchanged. A/B quality vs current path.
      Touch: completion_init_with_cached_chunks (optional skip question step),
      LlamaState runSingleRagQuery (fuse → prefillQuestionTail → decode),
      cacheBlendFuse(suffix_len: 0), buildStitchTokenList split (passage vs Q),
      timing metrics (separate “Question prefill” line).
  [ ] Q_sub gather in CHECK Step C (skip Q recompute on |imp|)
  [ ] enableHkvdProbe toggle in UI (default off for benchmarks)
  [ ] 10-question quality matrix: stitch vs full prefill vs stitch+GRAPH fuse
  [ ] Decode tuning: repetition penalty / lower max_tokens if verbose answers persist
  [ ] Optional: boundary recompute (last N tokens per chunk) before full fusion
  [ ] Content-addressed chunk cache (fix [n] index mismatch) —
      Today chunk_id = SHA256(model + "[n] passage…"), so the same passage at a
      different list position is a cache MISS (e.g. P3 as [3] vs [1]).
      TODO: hash raw passage text only; collect/save KV for body
      ("passage\n\n", no [n]); at stitch prefill "[n] " fresh (~2–4 tok/chunk)
      then merge cached body KV via seq_cp. Update ChunkStore.chunkId,
      ensurePassagesCached, completion_init_with_cached_chunks,
      buildStitchTokenList. Bump cache version / clear Documents/chunks after
      change (invalidates existing .bin keys). Matches reference CacheBlend
      (stable doc chunks) and improves reuse across Simple_test queries.
